# Отчёт по практикам 13-18

## Что было проделано

Проект был полностью переориентирован под ТЗ практик `№13-№18`.

Изначально в репозитории находилось приложение другой тематики, связанное с `auth`, `JWT`, `RBAC` и каталогом товаров. Эта логика не соответствовала заданиям из методичек `№13-№18`, поэтому проект был переработан в единое приложение заметок и напоминаний с PWA-функциональностью.

В результате были выполнены следующие доработки:

1. Переписан сервер в [backend/app.js](backend/app.js):
   - раздача статического клиента
   - маршруты `/api/config`, `/api/health`, `/subscribe`, `/unsubscribe`, `/snooze`
   - интеграция `Socket.IO`
   - отправка `push` через `web-push`
   - планирование напоминаний через `setTimeout`
   - поддержка локального HTTPS при наличии dev-сертификата

2. Полностью заменён фронтенд:
   - новый [frontend/index.html](frontend/index.html) как App Shell
   - новый [frontend/app.js](frontend/app.js) с логикой заметок, напоминаний, push и realtime
   - новый [frontend/sw.js](frontend/sw.js) для офлайн-режима и обработки действий уведомлений
   - новый [frontend/manifest.json](frontend/manifest.json)
   - новый [frontend/styles.css](frontend/styles.css)
   - новые контентные страницы [frontend/content/home.html](frontend/content/home.html) и [frontend/content/about.html](frontend/content/about.html)

3. Добавлены иконки PWA:
   - `32x32`
   - `128x128`
   - `192x192`
   - `512x512`
   - `maskable 512x512`

4. Удалены остатки старого React/Vite-приложения, чтобы структура проекта соответствовала текущему заданию и не вводила в заблуждение.

5. Подготовлен локальный HTTPS-скрипт [backend/scripts/create-dev-cert.ps1](backend/scripts/create-dev-cert.ps1) для Windows.

6. Переписан [README.md](README.md) под практики `13-18`.

## Теория по практикам

### Практика 13. Service Worker

`Service Worker` — это фоновый скрипт браузера, который работает отдельно от страницы и умеет:

- перехватывать сетевые запросы
- кэшировать ресурсы
- обеспечивать офлайн-режим
- обрабатывать push-уведомления

Ключевые события:

- `install` — первичное кэширование ресурсов
- `activate` — очистка старых кэшей и активация новой версии
- `fetch` — перехват запросов и выбор стратегии ответа
- `push` — получение push-сообщения
- `notificationclick` — обработка действий пользователя в уведомлении

В проекте эта теория применена через кэширование App Shell и работу приложения без сети после первого открытия.

### Практика 14. Web App Manifest

`Web App Manifest` — это JSON-файл с метаданными PWA. Он сообщает браузеру:

- как называется приложение
- какую иконку использовать
- какой стартовый URL открывать
- в каком режиме отображать приложение
- какие цвета применять при запуске

Важные поля:

- `name`
- `short_name`
- `start_url`
- `display`
- `background_color`
- `theme_color`
- `description`
- `icons`

В проекте добавлен полноценный `manifest.json`, иконки и `meta`-теги для корректной установки приложения на устройство.

### Практика 15. HTTPS + App Shell

`HTTPS` нужен для защищённой передачи данных и работы ряда браузерных API. В учебном контексте он важен прежде всего для PWA-возможностей и push-уведомлений.

`App Shell` — это минимальный каркас приложения:

- шапка
- навигация
- основной контейнер
- футер

Идея в том, что этот каркас загружается быстро и кэшируется отдельно, а содержимое страницы подгружается динамически.

В проекте:

- `index.html` играет роль App Shell
- страницы `home.html` и `about.html` подгружаются через `fetch`
- для каркаса используется `Cache First`
- для контентных страниц — `Network First`

### Практика 16. WebSocket + Push

`WebSocket` обеспечивает постоянное двустороннее соединение между клиентом и сервером. Это позволяет серверу отправлять события без повторных HTTP-запросов.

`Socket.IO` упрощает такую работу:

- подключение клиента
- отправка событий
- приём событий
- автоматическое переподключение

`Push API` позволяет присылать системные уведомления через браузер даже вне активной вкладки. Для этого используются:

- `PushManager`
- `Service Worker`
- `VAPID`-ключи
- библиотека `web-push` на сервере

В проекте:

- добавление заметки генерирует событие `newTask`
- сервер рассылает `taskAdded`
- во второй вкладке появляется realtime-toast
- при включённой подписке отправляется системное push-уведомление

### Практика 17. Детализация Push

На этом этапе push-система становится полезнее: уведомление связано не просто с фактом новой заметки, а с конкретным временем напоминания.

Основная логика:

1. Пользователь создаёт заметку с датой/временем.
2. Клиент сохраняет её в `localStorage`.
3. Клиент отправляет событие `newReminder` на сервер.
4. Сервер ставит `setTimeout`.
5. По истечении времени сервер отправляет push-уведомление.
6. Пользователь может нажать `Отложить на 5 минут`.
7. `Service Worker` вызывает маршрут `/snooze`.
8. Сервер переносит таймер.

В проекте это реализовано полностью, включая кнопку действия в уведомлении.
