## Практика 19. PostgreSQL

### Краткая теория

`PostgreSQL` — это реляционная СУБД. Реляционная база хранит данные в таблицах, где строки являются записями, а столбцы — полями. Такая модель удобна, когда структура данных заранее известна и должна строго соблюдаться.

Для работы с `PostgreSQL` в `Node.js` часто используются:

- драйвер `pg`
- ORM `Sequelize`

В проекте выбран `Sequelize`, потому что он позволяет описывать модель пользователя, создавать таблицу и выполнять `CRUD`-операции через JavaScript-код.

### Что реализовано в проекте

Для практики 19 добавлен отдельный набор маршрутов:

- `POST /api/postgres/users`
- `GET /api/postgres/users`
- `GET /api/postgres/users/:id`
- `PATCH /api/postgres/users/:id`
- `DELETE /api/postgres/users/:id`

Сущность пользователя содержит обязательные поля:

- `id`
- `first_name`
- `last_name`
- `age`
- `created_at`
- `updated_at`

Данные действительно сохраняются в `PostgreSQL`, а не в памяти приложения.

### Как проверить

## Практика 20. MongoDB

### Краткая теория

`MongoDB` — это документная `NoSQL` база данных. В отличие от реляционной базы, данные здесь хранятся не в таблицах, а в документах, близких по структуре к `JSON`.

Главное отличие `MongoDB` от `PostgreSQL`:

- схема данных гибче
- записи представляются как документы
- удобно хранить данные, структура которых может развиваться

Для работы с `MongoDB` в `Node.js` в проекте используется `Mongoose`, который позволяет описывать схему документа и выполнять операции чтения и изменения.

## Практика 21. Redis

### Краткая теория

`Redis` — это быстрое in-memory key-value хранилище. В учебной задаче оно используется как слой кэша между клиентом и основной логикой приложения.

Смысл кэширования:

- при первом запросе сервер получает данные обычным способом
- затем результат записывается в `Redis`
- повторный запрос получает те же данные быстрее, уже из кэша

Ключевые понятия этой практики:

- `TTL` — время жизни записи в кэше
- `cache middleware` — промежуточный слой, который сначала проверяет `Redis`
- `invalidate cache` — очистка кэша при изменении данных

### Что реализовано в проекте

В проекте реализовано учебное `RBAC`-приложение с `JWT`:

- `POST /api/auth/register`
- `POST /api/auth/login`
- `POST /api/auth/refresh`
- `POST /api/auth/logout`
- `GET /api/me`

Добавлены роли:

- `admin`
- `manager`
- `user`

Реализовано кэширование точно под идею методички:

- `GET /api/users` — `1 минута`
- `GET /api/users/:id` — `1 минута`
- `GET /api/products` — `10 минут`
- `GET /api/products/:id` — `10 минут`

Также реализована очистка кэша при изменении:

- пользователей
- товаров

Ответы кэшируемых маршрутов возвращают:

- `source: "server"` при первом запросе
- `source: "cache"` при повторном запросе в пределах `TTL`

### Как проверить

1. Поднимите `Redis`.
2. Откройте `/app`.
3. Выполните вход под `admin/admin123`.
4. Нажмите `Получить пользователей`.
5. Нажмите `Получить товары`.
6. Нажмите `Повторить товары для кэша`.
7. Убедитесь, что в блоке ответа меняется поле `source`.
8. Добавьте новый товар и снова запросите список товаров.
9. Убедитесь, что кэш был инвалидирован и ответ снова пришёл с `source: "server"`.

## Практика 22. Балансировка нагрузки

### Краткая теория

Балансировка нагрузки нужна для распределения входящих запросов между несколькими экземплярами одного приложения. Это помогает:

- не перегружать один сервер
- повышать отказоустойчивость
- масштабировать систему горизонтально

В методичке рассматриваются два программных балансировщика:

- `Nginx`
- `HAProxy`

Также важны параметры отказоустойчивости:

- `max_fails`
- `fail_timeout`

Они позволяют перестать направлять трафик на недоступный backend.

### Что реализовано в проекте

Для этой практики backend возвращает на `GET /` ответ с идентификатором сервера:

- `backend-1`
- `backend-2`
- `backend-3`

Добавлены локальные конфигурации:

- [infra/nginx/nginx.local.conf](infra/nginx/nginx.local.conf)
- [infra/haproxy/haproxy.local.cfg](infra/haproxy/haproxy.local.cfg)

Подготовлены отдельные команды запуска нескольких серверов:

- `npm run dev:backend1`
- `npm run dev:backend2`
- `npm run dev:backend3`

В `Nginx` реализованы:

- два основных backend
- один резервный `backup`
- параметры `max_fails` и `fail_timeout`

### Как проверить

1. Запустите `backend1`, `backend2`, `backend3`.
2. Подключите `Nginx` по локальному конфигу.
3. Выполните несколько раз:

```bash
curl http://localhost/
```

4. Проверьте, что поле `server` в ответе меняется.
5. Остановите один backend.
6. Повторите запросы.
7. Убедитесь, что балансировщик продолжает обслуживать трафик через доступные узлы.

## Практика 23. Docker

### Краткая теория

Контейнеризация нужна для того, чтобы приложение запускалось одинаково в любом окружении вместе со своими зависимостями. Для этого используются:

- `Dockerfile` — инструкция сборки образа
- `Docker Compose` — способ описать сразу несколько контейнеров в одном файле
- сети Docker — для связи контейнеров по именам сервисов
- тома Docker — для сохранения данных между перезапусками
- переменные окружения — для конфигурации без изменения кода

В проект добавлены:

- [backend/Dockerfile](backend/Dockerfile)

Через `docker-compose.yml` поднимаются:

- `postgres`
- `mongo`
- `redis`
- `backend1`
- `backend2`
- `backend3`
- `nginx`

Для `Nginx` подготовлен docker-конфиг:

- [infra/nginx/nginx.conf](infra/nginx/nginx.conf)

Также добавлен docker-вариант `HAProxy`:

- [infra/haproxy/haproxy.docker.cfg](infra/haproxy/haproxy.docker.cfg)
